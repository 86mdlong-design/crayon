0 Start here — the whole picture

What Stickr is, its pieces, its keys, and everything it talks to

Overview Supabase ↗ Stripe ↗ OpenAI ↗ Postgres ↗ 🔑 8 named keys

Explain it like I'm 5 5101520

Stickr is a little shop where people pin sticky notes. The shop has a front window (the page you see, public/), a counter person who answers every request (server.js), and two hallways behind the counter: one for notes and one for money. Notes are kept in a big filing cabinet (a database called Postgres). A helper company called Supabase remembers who paid for the fancy Pro plan. A payment booth (Stripe) takes the money, and a robot reader (OpenAI) can squeeze any note into one short sentence.

The inspector found three big problems: a master key taped to the front window, a money door that believes anyone who knocks, and note text glued straight into filing-cabinet instructions. The red dots in the index show where they live.

Stickr is a small web app: a plain HTML/JavaScript page in your browser, and a Node server behind it. The page never touches data directly — it sends requests to the server's API (its set of request-handling addresses, like /api/notes). The server keeps notes in a Postgres database, keeps each user's plan (free or pro) in Supabase, sends people to Stripe to pay, and asks OpenAI for one-line note summaries.

Secrets (keys and passwords) are supposed to live in a .env file the server reads at startup — the table below lists them by name. One of them also got pasted into browser code, which is the book's most serious finding.

Single-process Express monolith: serves the static frontend from public/, exposes a JSON API (/api/notes, /api/billing). Notes CRUD hits Postgres directly via pg; profile/plan state lives in Supabase (updated server-side with the service-role client); payments use Stripe Checkout plus a webhook that flips profiles.plan to pro; summaries call OpenAI chat completions. Config is env-driven via dotenv. There is no auth layer — every route trusts a caller-supplied userId.

Architecture notes: identity is client-asserted end-to-end (localStorage → query/body → SQL/Stripe client_reference_id), so every authorization property collapses; fixing auth is prerequisite to fixing anything else. Persistence is split across raw pg and Supabase Postgres — two sources of truth, one consolidation candidate. The webhook trust boundary is broken (no signature verification), and the service-role key exists in two places: env (correct) and shipped client bundle (fatal — rotate first). Tests cover pure utils only; the risky surfaces are untested.

2 .env.example

The labeled, empty envelopes for every secret the app needs

dotenv Supabase ↗ Stripe ↗ OpenAI ↗ 🟡 worth knowing

Explain it like I'm 5 5101520

Imagine a box of envelopes with labels but nothing inside: "Supabase master key goes here", "Stripe key goes here". This file shows every label so a new helper knows which secret notes the shop needs. The real notes go into a private copy named .env that must never leave the building. This example file is safe to share because the envelopes are empty.

A template for the real .env file. Each line is NAME=value; here every value is a placeholder. The server loads the real file at startup (via dotenv) so keys never sit in the code itself. The names tell you the app's whole supply chain: Supabase (two keys plus a URL), the Postgres address, two Stripe secrets, one OpenAI key, and the port number.

Standard dotenv template. Cross-checking names against the code: SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY → lib/supabase.js; DATABASE_URL → lib/db.js; STRIPE_SECRET_KEY → routes/billing.js; OPENAI_API_KEY → lib/ai.js; PORT → server.js. Two names are read by nothing: SUPABASE_ANON_KEY and STRIPE_WEBHOOK_SECRET. The second one is a bug wearing a config costume — see the billing chapter.

Unused env names are drift signals worth chasing: an unused ANON_KEY hints the frontend once talked to Supabase directly (it still carries the scar — see app.js); an unused WEBHOOK_SECRET means the verification step was planned and skipped. DATABASE_URL embeds credentials — treat the whole string as a secret. Operational rules: real .env stays gitignored, rotate anything that ever landed in history, and prefer per-env project refs over one shared Supabase project.

🔒 Attack surfaces that live here

  • This example file is safe, but the real .env it describes is a one-file jackpot. Make sure .env is listed in .gitignore before you ever fill it in — a secret committed once stays recoverable in git history even after you delete it.
  • STRIPE_WEBHOOK_SECRET is named here but no code reads it — the safety check it exists for is missing in routes/billing.js.

3 server.js

The front door — starts the server, sets house rules, opens the two hallways

JavaScript (Node) Express ↗ 🟡 security note below

Explain it like I'm 5 5101520

This file is the person who opens the shop every morning. It reads the secret notes (so the other helpers can use them), hangs up the front window page for visitors, and unlocks two hallways: /api/notes and /api/billing. It also keeps a little bell called /healthz that just rings "I'm alive!" One worry: it props the front door open for every neighborhood on the internet instead of just our own street.

The entry point — the file Node runs first. It builds an Express app and stacks up middleware: checks every request passes through, in order. cors({ origin: '*' }) answers "who may call this API from a web page?" with "anyone". express.json() unpacks JSON request bodies. express.static('public') serves the frontend files. Then it attaches the two routers and starts listening on the port from the PORT setting (or 3000).

Boot order: dotenv.config() before anything reads process.env (correct), middleware, routers, listen. The wildcard CORS matters because this API has no auth: any origin can fire requests that the notes router will happily honor with a caller-chosen userId. /healthz is a plain liveness probe. Nothing else lives here — the interesting logic is all in the routers.

Wildcard CORS is usually a 🟡 on its own; here it compounds the client-asserted-identity design into "any web page can read/write any user's notes from a victimless drive-by" — no credentials needed because the API never asks for any. Fix order: add real auth, then narrow origin to the app's own origins. Missing-but-standard: helmet, rate limiting, request size limits (express.json({ limit })), and a JSON error handler so async route failures don't fall through Express 4's default handler.

🔒 Attack surfaces that live here

  • CORS is open to every website (origin: '*') on an API that also does no login checks — any page on the internet can script calls against your data. Lock it to your own site's address once auth exists.

4 routes/notes.js

The notes hallway — list notes, pin new ones, ask the robot for a one-liner

JavaScript (Node) Postgres ↗ OpenAI ↗ 🟠 security notes below

Explain it like I'm 5 5101520

Three service windows in one hallway. Window 1: "show me the notes for this name" — and it hands over that pile without ever checking it's really you. Anyone who says "Bob" gets Bob's notes. Window 2: pins a new note in the cabinet. Window 3: reads one note to the robot and brings back a one-sentence version.

The scary part: windows 1 and 2 glue your words straight into the instructions they hand the filing cabinet. A visitor can write a note that isn't a note — it's a sneaky instruction the cabinet will obey, like reading someone else's drawer.

A router: a bundle of API addresses under /api/notes. The database speaks a language called SQL, and this file builds SQL sentences by pasting user text into them ("... WHERE user_id = '" + userId + "'"). If someone's "name" contains a quote mark, the sentence changes meaning — that's SQL injection. The third route shows the safe way: $1 is a fill-in-the-blank the database fills after understanding the sentence, so tricks can't rewrite it. Also: identity comes from whatever the caller typed (?userId= or the body), because there's no login system.

Three handlers on an Express Router. GET / and POST / interpolate userId, body, color straight into SQL — injection, twice. GET /:id/summary uses a parameterized query ($1) then pipes the note body to lib/ai.summarize. No schema validation, no length caps, no auth middleware; a quote in color doesn't just attack you, it 500s honest requests too. Note the asymmetry as your fix template: route 3's query style is what routes 1–2 should be.

Classic quote-breakout injection, reachable cross-origin thanks to the wildcard CORS upstream. The fix is boring and total: parameterize everything, validate shapes/lengths at the edge (zod or a 10-line checker), and derive userId from a session — which means building auth this app doesn't have. Two subtler notes: Express 4 doesn't catch rejected async handlers, so any db error here is an unhandled rejection (add an async wrapper or Express 5); and the summary route bills OpenAI per call on an unauthenticated endpoint — free-tier abuse vector.

The jobs inside

GET / handler
Fetches every note whose user_id matches the caller-supplied name, newest first.
POST / handler
Inserts a new note (owner, text, color) and returns its new id.
GET /:id/summary handler
Loads one note safely ($1), sends its text to OpenAI, returns the one-line summary.

🔒 Attack surfaces that live here

  • User text is glued into database commands in two routes — a crafted note or "name" can rewrite the command (SQL injection). Use fill-in-the-blank parameters ($1) like the summary route already does.
  • Identity is whatever the caller claims: any visitor can read or write any user's notes by changing userId. Only a real login system fixes this.
  • No input checks — a 10-megabyte "note" or a nonsense color is accepted as-is and passed to the database and to OpenAI (which charges per word).

5 routes/billing.js

The money hallway — sends people to pay, and hears (too trustingly) that they paid

JavaScript (Node) Stripe ↗ Supabase ↗ 🔴 security notes below

Explain it like I'm 5 5101520

Booth 1 hands you a golden ticket to Stripe's payment store, with your name written on it — except it writes whatever name you tell it. Booth 2 is a back door where Stripe knocks to say "someone paid!" The problem: the booth never checks who's knocking. Anyone can knock, say "Bob paid!", and the shop makes Bob (or themselves) a VIP for free. Stripe gives shops a wax seal to check exactly this — our booth never looks at it.

Two routes. POST /checkout asks Stripe to create a Checkout session — a Stripe-hosted payment page — and sends back its link; the caller's userId rides along as client_reference_id so we know who to upgrade later. POST /webhook is Stripe's callback: when a payment finishes, Stripe POSTs an event here. Stripe signs every event, and apps are supposed to verify that signature with STRIPE_WEBHOOK_SECRET. This code skips the check and believes any JSON that arrives, then sets that user's plan to pro in Supabase.

checkout: stripe.checkout.sessions.create with a hardcoded price id, redirect URLs to localhost, and body-supplied client_reference_id. webhook: reads req.body (already JSON-parsed by the global express.json()) and matches event.type === 'checkout.session.completed'. The correct pattern needs two changes at once: mount express.raw() on this route (verification needs the exact bytes), and call stripe.webhooks.constructEvent(rawBody, req.headers['stripe-signature'], STRIPE_WEBHOOK_SECRET) — which throws on forgeries. On success it updates profiles.plan via the service-role Supabase client.

Two flaws compound: forgeable events × attacker-chosen client_reference_id = anyone can mint Pro for any account with one curl. Beyond constructEvent: dedupe on event.id (webhooks redeliver), don't trust event payload fields for money decisions — re-fetch the session/subscription from Stripe's API, and return 2xx fast to avoid retry storms. Hardcoded price_… id and localhost URLs belong in env. The service-role update is the right tool here; it's the unverified trigger that's wrong.

The jobs inside

POST /checkout handler
Creates a Stripe Checkout session for the Pro subscription and returns its payment link.
POST /webhook handler
Hears "payment finished" events and flips that user's plan to pro in Supabase — currently without proving the message came from Stripe.

🔒 Attack surfaces that live here

  • The webhook never verifies Stripe's signature — anyone who finds /api/billing/webhook can fake a "payment succeeded" message and get Pro for free. STRIPE_WEBHOOK_SECRET exists in the config for exactly this check; the code never uses it.
  • The checkout ticket's identity (client_reference_id) is whatever the caller sends — combined with the flaw above, an attacker can upgrade (or target) any account, not just their own.

7 public/app.js

The page's brain — and the window with a master key taped to it

JavaScript (browser) Supabase ↗ 🔴 security notes below

Explain it like I'm 5 5101520

This is the helper elf that rides along in every visitor's pocket. It fetches your notes from the counter, pins new ones, and walks you to the payment booth when you press Go Pro.

Two scary things live here. First: someone taped a master key to the front window — a secret called SUPABASE_SERVICE_ROLE_KEY is written right in this file, and everything in this file is handed to every visitor. A master key opens every drawer in the filing company, ignoring all the locks. (The book never repeats the key itself — but it's there.) Second: when the elf pins notes to the board, it treats the note's words as real instructions, not just words — so a nasty note can boss the page around when anyone reads it.

Browser JavaScript: every visitor downloads this file, so nothing in it is secret. Yet it hardcodes a Supabase service-role key — the kind that bypasses every data-protection rule. The code never even uses it (a leftover "TODO: move somewhere safer"), but unused doesn't mean safe: it shipped, so treat it as stolen and rotate it. Second issue: div.innerHTML = note.body pastes note text as live HTML — if a note contains a script tag, it runs in the reader's browser (called XSS). The fix is textContent, which pastes words as words. Third: "who am I" comes from localStorage, a self-written name tag anyone can edit.

Three fetch wrappers (loadNotes, addNote, goPro) against the local API, wired to the ids from index.html. The hardcoded SUPABASE_SERVICE_ROLE_KEY constant is dead code — no supabase-js in the browser — which is exactly how leaks happen: a refactor moved the calls server-side and left the credential in the bundle. Rotate it in Supabase, then delete the constant. The XSS is stored: the POST route saves raw text, this file renders it with innerHTML, so one poisoned note fires for every reader of that board. userId from localStorage feeds the API's client-asserted identity problem end-to-end.

Triage order: (1) rotate the service-role key now — bundle shipped = leaked, and git history keeps it; (2) innerHTML → textContent (one line) plus server-side length/shape validation, with CSP as belt-and-braces; (3) replace localStorage identity with a real session (Supabase Auth or a signed cookie) and derive the user server-side, which is the same fix ch-4 needs. Note the shape of the failure: the unused key + TODO comment is the classic vibe-code artifact — secrets migrate to the server, their copies don't get deleted.

The jobs inside

loadNotes()
Asks the API for this user's notes and pins each one to the board (currently as live HTML — the XSS).
addNote()
Sends the typed note to the API, clears the box, reloads the board.
goPro()
Asks the billing hallway for a Stripe payment link and sends the visitor there.

🔒 Attack surfaces that live here

  • A Supabase service-role key is hardcoded in code that ships to every visitor's browser. It bypasses all row-level protection. Even though nothing here uses it, it's public now — rotate it in Supabase and delete the constant. (Named only — this book never prints values.)
  • Notes render via innerHTML, so a saved note containing markup runs in every reader's browser (stored XSS). Use textContent.
  • Identity is a self-written localStorage name tag — it feeds the "trust whatever userId arrives" problem in the notes API.

10 lib/ai.js

The robot reader — mails a note to OpenAI, gets one sentence back

JavaScript (Node) OpenAI ↗ 🟡 worth knowing

Explain it like I'm 5 5101520

When someone asks for the short version of a note, this file mails the whole note to a robot in another city (a company called OpenAI) and waits for one sentence back. Each letter costs a few pennies, paid with the OPENAI_API_KEY secret note. Worth remembering: the note leaves the shop — a stranger company reads it, even if it's your grocery list… or your diary.

Sets up the OpenAI client with a key from the environment (good — not hardcoded) and exports one function: summarize(text). It sends a two-part message — a system instruction ("summarize in one short sentence") and the note text — to the gpt-4o-mini model, caps the reply length with max_tokens: 60, and returns the trimmed sentence. Every call costs real money, which is why "who may trigger this?" matters.

Minimal chat-completions wrapper. What it lacks at this size: a timeout/abort (a stalled OpenAI call currently hangs the route), retry policy, and an input length cap — note bodies arrive unvalidated from ch-4, so cost scales with whatever a caller pastes. Called only from GET /:id/summary. The system-prompt/user-content split is correct; treat note text strictly as content, never as instructions.

Ship-blockers none, hardening cheap: pass a request timeout, clamp input (e.g. first 2–4k chars), catch and map provider errors to a 502 instead of an unhandled rejection, and consider caching summaries per note id. The interesting risks are systemic: unauthenticated callers can run up token spend (free-abuse vector via ch-3/ch-4), and note content egresses to a third party — that's a privacy-policy line item, not just an engineering one.

Three cases, all for utils/format.js. Tests are a safety net: they catch it when a future change quietly breaks old behavior — but only for code they cover.

Straightforward node:test + assert/strict usage, no mocks needed since the targets are pure. Coverage honesty: the only tested module is the one module the app doesn't use. The routes — injection, webhook, auth behavior — have zero regression protection, which is where tests would pay for themselves.

Highest-value next tests, in order: notes routes via supertest (assert parameterized queries reject quote-breakout inputs once fixed), webhook route with forged vs. correctly-signed fake Stripe events (signature check as an executable spec), and a length-cap test on the summary path. The timeAgo "fresh timestamp" test has a theoretical flake window if the clock ticks between ISO generation and comparison — irrelevant at < 60s granularity.
